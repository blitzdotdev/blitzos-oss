# CLAUDE.md — BlitzOS

Guidance for Claude Code working in the BlitzOS repo.

## What this is (V1 = island-only)

BlitzOS is an Electron macOS **dynamic island** (the notch) for human+agent collaboration. The human drives it, and so can an AI agent over the [agent-socket](https://agentsocket.dev) relay (paste a URL / "connect to blitz os", no MCP) or the localhost control server. **The island IS the whole UI** — there is NO canvas, no infinite plane, no surfaces-on-a-canvas, no pan/zoom, no slot lattice, no stages, no workspace switcher, no dock/Sidebar, no Overview, no radial menu, no folders/annotations. (All of that was the pre-V1 desktop; it was cut. See "what got cut" under Gotchas.)

**The island.** A black pill at the notch that opens — on hover, or ⌥Space — into a body-portal chassis: hover shows a **home grid of widget icons** (V1 ships ONE functional icon, **Chat**, flanked by dotted placeholders for the agent-made widgets that come post-V1); ⌥Space just shows/hides the island (restoring the last view+tab), never spawning or mutating state. Chat is the agent session UI: a tab strip (a pen button that spawns a brand-new agent and enters it, plus one tab per live agent — Blitz '0' first), an iMessage-style transcript interleaved with the narrator's plain milestone steps, a live status line, a steer bar, a "Details" expand for raw tool rows, and a "+" attach panel for **connections** (browser/computer use). Web surfaces are GONE — the agent works the user's REAL browser (the connector extension) and native apps (the computer-use helper) instead. **Widgets are DEFERRED** (experimental, post-V1): the agent generating/pinning its own island widgets and apps is not in V1, which ships only Chat.

**Agent-runtime model (why BlitzOS exists).** BlitzOS is an *OS for an agent*: it turns ANY connected agent (Claude Code today over agent-socket, any agent, a built-in chat client later) into an autonomous one with **zero per-task code**. The agent supplies intelligence; BlitzOS supplies the loop. Four parts: **syscalls** (the agent's tools/hands — chat, connections, terminals, agents, workflows), **perception** (a content-agnostic world stream, the agent's eyes), **a scheduler** (coalesced "moments" that *wake* the agent with a snapshot, the interrupt), and the **agent as swappable policy** (it decides significance and action). The whole point is **out-of-distribution generalization**: perception is dumb-but-rich and the agent decides what matters, so a new task (coach my chess, draft this email, summarize this PDF) needs no new BlitzOS code. **Never hand-build a per-task watch loop** — make perception and wake general, and let the agent's policy handle the task. See "Agent runtime" under Architecture.

Prototype stage (version 0.0.1); the island/runtime agent doctrine lives in `src/main/blitzos-agents.md` (match its island framing).

## Commands

```bash
npm run dev        # electron-vite dev (the GUI; macOS only) — predev ensures the native helpers
npm run build      # electron-vite build → main + preload + renderer to out/
npm run typecheck  # tsc --noEmit -p tsconfig.json
npm run check      # typecheck + parity (scripts/check-parity.mjs) + build — the green-it gate
npm run dist       # package BlitzOS.app (signed+notarized when ~/.zshrc Apple creds present) -> release/*.zip
```

There is **no display in CI / headless sandboxes**, so you cannot see the GUI. Verify behavior instead by: launching `npm run dev > /tmp/aos.log 2>&1`, then reading the log for `did-finish-load`, the printed control-API token, and the agent-socket paste URL; and by driving the app via the control API or agent-socket tools (below) and checking `list_state`. Never claim the pixels look right, that's the user's to confirm.

## Stack & layout

electron-vite + React + TypeScript + zustand.

```
src/main/        Electron main (Node)
  index.ts         the one BrowserWindow, notch overlay, boot-task seam, wires everything
  notch-overlay.ts the dynamic-island overlay window + the always-interactive notch hit-window
  osActions.ts     control plane: IPC mutations + getState, the live-WebContents registry
  control-server.ts  localhost HTTP control API (trusted local-agent path; mints session.json)
  agentSocket.ts   connects to the agent-socket relay; tools -> osActions (remote agent path)
  os-tools.mjs     THE one shared agent-tool registry (both transports — see Architecture)
  agent-runtime.mjs / terminal-manager.mjs   the managed-agent backends + tmux terminals
  events.ts / perception-core.mjs   the perception kernel → moments → /events wake
  blitzos-agents.md   the runtime agent's island doctrine (what an agent reads on connect)
src/preload/index.ts   contextBridge api: onAction, sendState, onAgentSocketUrl, notch.*
src/renderer/src/
  App.tsx          the notch wiring + body-portal of the island; NO canvas render
  notch/           THE island UI (KEPT): NotchHost / IslandHome / IslandPanel / ChatInput / AttachPanel
  store.ts         zustand — runtime panels (chat/terminal/inbox) + viewport (mid-migration, see Gotchas)
  components/{ConnectPicker,Icons}.tsx
vendor/agent-socket-sdk/   vendored @agent-socket/sdk dist (see "Gotchas")
```

## Architecture

- **The island (the whole UI).** `App.tsx` runs ONE `BrowserWindow` reconfigured as the notch overlay (`notch-overlay.ts`): a transparent, all-Spaces, full-display window whose `#root-canvas` is clipped to the macOS notch shape (a rounded-bottom pill — the dynamic-island look). There is no canvas behind it; the clip never grows past the pill. The island UI is rendered OUTSIDE that clip as **body portals** (`createPortal(..., document.body)`): the `notch-handle` pill (always the top hit-target) plus the `NotchHost` chassis. `notch/` is the island React tree, and it is **canvas-independent** (imports nothing from `store`/canvas):
  - `NotchHost.tsx` — the stateful shell: owns the active page (1..N = agent at page-1; Blitz '0' is page 1; the pen spawns a new agent immediately, not a deferred composer), tab nav (Ctrl+Tab over the agent tabs), and wraps everything in the invariant black `.nh-chassis` (grows wide when the attach panel opens). Pulls a one-shot agent snapshot on open, then rides the live `os:action {type:'chat'}` broadcast.
  - `IslandHome.tsx` — the hover **home grid**: a row of widget icons, V1 = one Chat icon (centered, live-pulse when an agent is working) flanked by dotted placeholders. Tap Chat → Blitz `'0'`'s session (`view:'session'`), never a new spawn.
  - `IslandPanel.tsx` — the session UI: the tab strip, the active agent's transcript (iMessage bubbles + the narrator's milestone steps), a status line, the steer bar, a "Details" expand for raw tool rows, and a "+" that toggles `AttachPanel.tsx` (connections) inline.
  Open paths: **hover** (or the notch hit-window's reported hover) → `home`; **⌥Space** → show/hide the island (pinned open), restoring the last view+tab; it never spawns a session or mutates state. The hit-window drives the toggle/hover so the notch is clickable in every state with no click-through race. Esc closes the island.
- **Control plane.** `osActions.ts` is the single source of truth for OS mutations. Both the **localhost control server** (HTTP on 127.0.0.1, trusted) and **agent-socket** (relay, untrusted) call it; it sends `os:action` IPC to the renderer, which applies it to the zustand store, and the renderer pushes `os:state` back so `list_state` works. The agent↔OS contract is plain HTTP/JSON, **not MCP** (deliberate). The control server mints `~/.blitzos/session.json` (a `local:{url,token}` block + the relay url) — a co-located agent reads it for the trusted localhost path; agentSocket.ts mints the relay paste URL (surfaced via the in-app "Connect AI" button).
- **Agent tools (the syscalls).** **Defined ONCE in `src/main/os-tools.mjs`** (`makeOsTools(ops)`) — the single shared registry for BOTH transports: the relay path (`agentSocket.ts`) and the localhost control path (`control-server.ts`) both bind it via `electron-os-tools.ts` (`electronOps`), so the two paths can never diverge. To add/change a tool, edit `os-tools.mjs` once. Representative V1 syscalls: `say` / `ask` (talk to the user in chat), `/events` (the wake loop), `steer` (wake another agent), `spawn_agent` / `close_agent` / `rename_agent` (peer agents), `open_terminal` / `send_to_terminal` / `read_terminal` / `close_terminal` / `remove_terminal` / `list_terminals` (real tmux terminals), `start_workflow` / `run_workflow` / `set_orchestrators` (blitzscript workflows), `request_action` / `list_actions` / `resolve_action` (the human action-items inbox), the `connection_*` and `blitz_chrome_*` families (the user's real browser/app — see Connections), and `new_app` (provision a real blitz.dev deliverable). The canvas/web/widget tools (`create_surface`, `open_window`, `update_surface`, `close_surface`, `surface_control`, `read_window`, `get_surface`, `get_system_ui`, and the widget tools) were CUT in V1; **see `os-tools.mjs` for the authoritative set** (~50 tools). The underlying `ops` (createSurface etc.) stay — connections + run_workflow call them internally.
- **Connections (the V1 integration — browser + computer use).** The agent acts in the user's accounts by driving what the user CONNECTS into BlitzOS: a Chrome/Safari **tab** (via the BlitzOS Connector extension, `extension/`, force-installed by `connection-install.ts`) or any macOS app **window** (via the computer-use helper). There is no token API and no BlitzOS-owned `web` surface — the user's own logged-in browser IS the integration. A connection is a per-source TOOL PROVIDER (`connection-ops.mjs` + `connection-{tab,safari,window}-link`): `connection_read` (a tab's DOM/text, a window's AX tree, or a screenshot), `connection_act` (click/type/set by ref), `connection_run_js` (tab only), and `connection_save_tool` (persist a derived op per `sourceId`, inherited by every future connection to that source). Auth is browser-first: the user's session is the real auth; the agent confirms the signed-in identity (`connection_read`) before any outward action.
- **Agent runtime (perception → moments → wake).** The autonomy half (see "Agent-runtime model" above) lives in the perception kernel `events.ts` (Electron) / `perception-core.mjs` (shared, also the server backend). Raw signals funnel through `emit()`; the *coalescer* batches them into **moments** — framed snapshots `{trigger, signals, user[], snapshot?, url?, title?}` — emitted on a ~15s cadence OR immediately on a significant transition. `/events {since, wait}` long-polls moments; the agent runs ONE loop and is **woken** per moment, never per keystroke. **V1 wake sources:** `message` (island chat), `connection` (the user's real browser/app changing or (dis)connecting — now the PRIMARY world signal), `tick` (the status-only supervisor heartbeat, below), and `system` (e.g. a crash announced after an unclean shutdown). **Cut with the canvas:** `trigger:'canvas'` (surface geometry) and `'annotation'` are gone, and the in-page `INJECT`/`DRAIN` DOM sensors survive ONLY for the server-mode headless Chromium — the Electron island has no `web` page to inject into. **Generalization rule:** keep BlitzOS perception content-agnostic; the agent interprets significance and acts. No per-task detection (e.g. no "game over" logic) belongs in BlitzOS.
- **The supervisor tick (status-only in V1).** `emitTick` (perception-core) snapshots the host world from a transport-registered source (`setTickSource`), diffs it against the prior tick, and emits ONE `trigger:'tick'` moment IFF the diff is material — waking the primary agent `'0'` (the supervisor) to `/steer` a stalled/erred/diverged worker. **V1 diffs ONLY agent-status edges + terminal exits** (`getTickSnapshot` drops everything but agent status + terminal state — no surface geometry, there is no canvas). Materiality is content-agnostic transition-shape only (a status edge, a terminal exit, an agent added/closed); ZERO per-task/threshold heuristics live here. A quiet world emits nothing. `/steer` is the redirect mechanism (the island UI's steer bar = "message this agent"). **Deferred (post-V1):** on each wake `'0'` also reading the woken worker's session `.jsonl` to judge whether it is actually on-task (Deep supervision).
- **Boot-task seam (onboarding + the primary agent's duty).** `agent-runtime.mjs` exposes a policy-free `setBootTaskProvider` → `getBootTask`: an optional duty string re-read on every (re)launch and injected into an agent's bootstrap with a license to act unprompted. `index.ts` wires it so ONLY the primary agent `'0'` gets a duty: `interviewBootTask()` (from `onboarding.ts`) returns the **chat-only** onboarding/interview duty while it is pending, then the resident initiative duty after. There is no case-file board and no seeded widgets — the whole onboarding flow happens in ONE agent chat (the scan survives only as the agent's context primer). Duty docs: `src/main/blitzos-interview.md`, `blitzos-onboarding.md`, `blitzos-orchestrator.md` (all chat-only).

## Gotchas / conventions

- **NO zustand. Use a native React external store.** Do NOT add zustand (it is unoptimized) for any NEW state. For state that must live OUTSIDE a component (shared, or surviving a remount), write a tiny module-level external store + `useSyncExternalStore` — `src/renderer/src/notch/stagingStore.ts` is the reference pattern: a module `let` holding the data, a `Set` of listeners, mutator fns that replace only the changed slice (so `getSnapshot` returns a stable ref and only affected subscribers re-render), and a `useX()` hook wrapping `useSyncExternalStore`. The legacy canvas `store.ts` (`useDesktop`) is the ONLY zustand store and is being phased out — do not extend it or copy it. Never persist UI state by mirroring it into a remounting component (seed-on-mount + write-through is fragile: a mount-time effect wipes the seed). Put it in the store, which never remounts.
- **Design quality bar.** UI you build (the island chassis, the home grid, the session/chat UI) must match the real BlitzOS design, not generic AI slop. Read the design tokens (`src/renderer/src/tokens.css`) and match them. When fixing a visual flaw, fix every instance (all corners, all overflow, all intersecting chrome), not only the one called out.
- **The SDK is ESM-only; Electron main is CJS.** So `@agent-socket/sdk` is **bundled** into main via `externalizeDepsPlugin({ exclude: ['@agent-socket/sdk'] })` in `electron.vite.config.ts`, not `require()`d. `ws` stays external.
- **The SDK is vendored** at `vendor/agent-socket-sdk/` (source: github.com/repalash/agent-socket, Apache-2.0). To update: rebuild the SDK from source and re-copy `dist` + `package.json` into `vendor/agent-socket-sdk/`.
- **Never commit secrets.** `node_modules/`, `out/`, and `release/` are gitignored. Verify before any commit.
- **Persistence (the `.blitzos` journal).** State survives a restart in the workspace folder: `<dir>/.blitzos/workspace.json` (still `{camera, mode, stack, nodes[]}` from the desktop era — camera/mode are dead fields now), the chat transcripts in the workspace root (`chat.md` for `'0'`, `chat-N.md` for peers; `index.ts`'s chat.md tail emits reply lines + status), terminals under `.blitzos/terminals/<id>/` (tmux-backed, resumable), and the onboarding inputs under `.blitzos/onboarding/`. Runtime-only panels (chat/terminal/inbox) are reconstructed on boot, not persisted as nodes. **V1 targets ONE implicit workspace** — there is no UI switcher and no agent `switch_workspace` tool — but the multi-workspace machinery (`workspace-host.mjs`, `osListWorkspaces`/`createWorkspace`/`switchWorkspace`) still exists underneath; collapsing it to a single store (cut-plan step 8) is not finished.
- **Off-screen liveness:** `backgroundThrottling: false` on the window so the agent keeps running while the island is collapsed.
- **The canvas was cut in V1.** Everything that rendered surfaces on an infinite plane — the camera/`cameraController`, the home slot lattice (`stage-core`/`stages-core`), `SurfaceFrame` + the four `web`/`app`/`srcdoc`/`native` renderers, `WebTabView`/`BrowserNav` (the `<webview>` browser), the Sidebar/dock, Overview, the radial menu, folders/annotations, and the onboarding case-file board — is deleted. Don't re-introduce a canvas concept; V1 is the island.
- **The notch overlay** (`notch-overlay.ts`) is the ONE window. It is reconfigured transparent, all-Spaces, full-display; the renderer clips `#root-canvas` to the notch pill (`os:notch-interactive` toggles click-through as the renderer reports hover). The physical handle + the island chassis are body-portals z-stacked ABOVE the clip (App.tsx) so the notch stays the top hit-target. In overlay mode the native traffic lights are hidden and App.tsx draws its own; "fullscreen" is a no-op (an all-Spaces overlay can't native-fullscreen). An always-interactive transparent **hit-window** sits over the physical notch and drives the click/hover so it works in every state with no click-through race (the native notch geometry comes from `native/notch-geometry`). Tests: `node scripts/test-notch-hit-window.mjs`.
- **Attach-close hold heuristic (DO NOT REMOVE).** When the attach panel goes ON→OFF, the island is held open for ~1.5s (`NOTCH_ATTACH_CLOSE_HOLD_MS` in `App.tsx`, stamped onto `notchHoldUntilRef` in `onIslandAttachChange`) even if the cursor has already left the chassis, so collapsing the attach panel never yanks the island shut under the user. This has been silently dropped by a merge before; if you touch the notch hover/close logic, preserve it.
- **Popup policy** (`popup-policy.mjs` + `attachGuestWindowPolicy`, attached per guest in `did-attach-webview`): still wired for any remaining browser guest (e.g. an OAuth login popup) — sized `features` → real window, `about:blank` → hidden child, link disposition → new surface, else deny + swallow the `top.location` hijack. Mostly vestigial in V1 (no `web` surface), kept because the connector path can still spawn guests. Never "allow as hidden window" for URLs that fail top-level — that churn use-after-free'd Electron 31's guest-view manager (the 2026-06-09 SIGSEGV).
- Relay default is `agentsocket.dev` (`aisocket.dev` = same Worker); override with `AGENT_SOCKET_RELAY`.
- **Computer Use helper** — the computer-use connection's backend. Accessibility + Screen Recording TCC grants would force the GRANTED app to quit+reopen, fatal mid-session, so they DON'T live on BlitzOS: a separate Developer-ID-signed native helper **`BlitzComputerUse.app`** (`dev.blitz.os.computeruse`, `LSUIElement`, `native/computer-use-helper/`, built+signed by `scripts/dist-mac.sh`) HOLDS them. The load-bearing trick: the helper is launched via **LaunchServices** (`open -n`), making it its OWN responsible process with its OWN TCC identity — a child we spawn would inherit BlitzOS's identity. IPC = a Unix socket BlitzOS owns + newline-JSON; liveness = the socket. "Quit & reopen for the grant to take effect" = relaunch the HELPER (`relaunchForGrant`), BlitzOS untouched. Lifecycle in `src/main/computer-use-helper.ts` (install to `~/Library/Application Support/BlitzOS` for a stable bundle the grant sticks to, supervise, RPC); a connection to a macOS window (`connection-window-link.ts`) drives it. Tests: `node scripts/tests/test-computer-use-helper.mjs`. Dev caveat: TCC identity separation is only real in a signed packaged build; dev verifies build/launch/socket/relaunch mechanics.
- **Kernel fault model:** one instance per machine (`requestSingleInstanceLock`; a second launch focuses the first). `<root>/.blitzos/state.json` is the runtime journal: a boot after an unclean shutdown announces the crash to the human (chat line) + the agent (`trigger:'system'` moment, kind `crash`), enriched from macOS DiagnosticReports when possible. `markClean` runs LAST in `before-quit`/`gracefulExit` — "clean" means state flushed first. Two hosts on one root are detected via the journal's pid+heartbeat (loud warning today). Tests: `node scripts/tests/test-root-state.mjs`.
- **Effect-verified syscalls (`ok` = it happened).** The live V1 verification path is `connection_act` (and saved `act` tools): it returns the observed `effect` (the value typed, the DOM/AX change) so the agent confirms the act landed in-band, and a saved tool that comes back stale/empty means its selector rotted (re-derive + re-save, never report the stale result). A saved `act` tool that does not return its effect is the enemy (a silent no-op).
- **Guest capability contract** (`guest-capabilities.ts` + pure `popup-policy.mjs`): for any browser guest the connector spawns, browser-initiated escapes are handled by **what kind of action it is, never which site**. Downloads → stream into the workspace folder. Permissions → a real browser-parity **Allow/Block** prompt (`consent-card` in App.tsx), remembered per-origin in the root journal (`getPermission`/`setPermission`; sensitive set prompts, harmless auto-allowed, unknown denied). `beforeunload` → auto-accept so close/nav always wins. Session-level handlers (download/permission) set once on `persist:agentos`. Tests: `node scripts/tests/test-popup-policy.mjs`.
- **Onboarding is chat-only (no board).** First launch is gated by `ONBOARDING_MODE` in `src/renderer/src/onboarding/config.ts`; `OnboardingFlow.tsx` shows the opener (aurora + real scan progress from `scripts/onboarding-scan.mjs`). There is NO deterministic case-file board and NO seeded widgets — the whole flow (scan → interview → personalization) happens in ONE agent chat (agent `'0'`). The scan survives only as the agent's **context primer** (`.blitzos/onboarding/{scan.json,context.md}`). The duty is delivered via the boot-task seam (above): `interviewBootTask()` (`onboarding.ts`) returns the interview duty while `interview.json` is `pending`, then the resident initiative duty once `done`. Duty docs `src/main/blitzos-interview.md` + `blitzos-onboarding.md` are chat-only and de-externalized: the agent reads `context.md`, asks ≤4 MC questions as ```blitz-ui choice cards in chat + one open voice sample, finishes with a "What I learned" + `profile.md`, then proposes/starts one reversible initiative within the user's act-vs-ask boundaries. No canned/static interview tier; if no agent backend is available a plain chat error says the interviewer can't start.
